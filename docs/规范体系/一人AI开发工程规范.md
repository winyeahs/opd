# 一人AI开发工程规范

**文档版本**：V1.0  
**发布日期**：2026-02-27  
**文档性质**：规范性文档  
**适用范围**：所有采用一人AI开发模式的软件项目

---

## 一、核心概念

### 1.1 设计背景

**问题**：个人开发者资源有限，AI辅助开发缺乏系统性规范

**解决方案**：建立标准化的六阶段开发流程体系

**核心价值**：效率提升3-5倍，获得规范化开发能力

### 1.2 开发者角色

一人AI开发模式下，开发者需承担多个角色职责：

- **需求主导**：定义业务需求、确定项目范围
- **决策中心**：关键技术决策、优先级排序
- **资源管理**：时间规划、任务分配、风险控制
- **质量把关**：最终审核、验收交付

## 二、开发流程

### 2.1 六阶段流程

| 阶段 | 名称 | 占比 | 核心任务 |
|------|------|------|----------|
| 阶段一 | 需求与规划 | 10-15% | 需求分析、项目计划、验收标准 |
| 阶段二 | 设计与架构 | 15-20% | 架构设计、技术选型、接口设计 |
| 阶段三 | 开发与实现 | 40-50% | 代码开发、单元测试、代码审查 |
| 阶段四 | 测试与验证 | 15-20% | 测试设计、测试执行、缺陷管理 |
| 阶段五 | 部署与交付 | 5-10% | 部署配置、发布说明、用户手册 |
| 阶段六 | 运维与迭代 | 持续 | 系统监控、问题分析、迭代优化 |

**【方法论】六问题强迫思考**

需求分析阶段采用"六问题强迫思考"方法论，在编写需求文档前必须回答以下六个问题：

| 问题编号 | 问题内容 | 目的 |
|----------|----------|------|
| Q1 | 用户现在的痛点是什么？具体例子，不是假设 | 挖掘真实需求，避免伪需求 |
| Q2 | 用户目前是怎么解决这个问题的？ | 了解现状，找到改进空间 |
| Q3 | 如果不解决这个问题，后果是什么？ | 确认需求优先级和紧迫性 |
| Q4 | 你提出的功能真的解决了痛点吗？还是只是"听起来不错"？ | 挑战方案有效性 |
| Q5 | 最小可行方案是什么？明天能发布的是什么？ | 确定MVP范围，避免过度设计 |
| Q6 | 这个需求的假设前提是什么？假设成立吗？ | 检验需求根基，避免在错误假设上建设 |

**执行要点**：
- 每个问题必须给出具体答案，不得跳过
- 答案要有具体例子支撑，拒绝泛泛而谈
- 如果某个问题无法回答，说明需求分析不够深入
- 通过六问题后，需求规格说明书才能进入下一阶段

**【方法论】真实浏览器QA**

测试阶段采用"真实浏览器QA"方法，使用 Playwright 等工具打开真实浏览器进行测试验证：

**测试流程**：
1. **启动浏览器**：使用 Playwright 打开 Chrome/Firefox/Safari 等真实浏览器
2. **访问测试环境**：连接到 staging 或本地测试服务器
3. **执行用户流程**：按照测试用例，模拟真实用户操作（点击、输入、导航）
4. **截图记录**：关键步骤截图留存，便于问题追溯
5. **自动修复**：发现问题后，AI 自动分析并修复代码
6. **回归验证**：修复后重新执行测试流程，确保问题解决

**优势**：
- 发现纯单元测试无法捕获的UI/交互问题
- 验证真实用户体验，而非模拟环境
- 自动化执行，支持 CI/CD 集成
- 截图和日志可作为测试报告证据

## 三、操作指南

### 3.1 链式调用机制

六阶段开发流程采用**链式调用**设计：每个阶段的产出文档自动成为下一阶段的输入，形成闭环。

```
需求规格说明书 → 架构设计文档 → 技术实现 → 测试用例 → 发布说明 → 迭代计划
        ↓              ↓              ↓          ↓          ↓
      验收标准 ← ← ← ← ← ← ← ← ← ← ← ← ← ← ← ← ← ← ← ← ← ← ← 
```

**核心原则**：
- **上游产出、下游读取**：阶段一的需求规格说明书直接被阶段二的架构设计读取
- **文档即契约**：每个交付物是上下游之间的"契约"，下游不得偏离上游定义
- **闭环验证**：最终产出回溯验收标准，确保不偏离原始需求

**文档流转关系**：

| 上游文档 | 下游消费方 | 消费方式 |
|----------|------------|----------|
| 求规格说明书 | 架构设计、接口设计 | 直接读取，提取功能点 |
| 架构设计文档 | 代码实现 | 按架构约束编码 |
| 测试用例 | QA执行 | 自动化/手工测试 |
| 测试报告 | 发布决策 | 决定是否可发布 |
| 发布说明 | 运维监控 | 建立监控指标 |
| 运维报告 | 迭代计划 | 形成下一周期输入 |

### 3.2 执行流程

1. **明确目标**：确定当前阶段和任务目标
2. **读取上游**：读取上一阶段的产出文档作为输入
3. **执行任务**：按照阶段要求执行具体任务
4. **产出文档**：生成当前阶段的交付物
5. **审核验收**：检查成果是否符合质量门禁
6. **阶段切换**：交付物作为下一阶段输入，继续执行

### 3.3 执行模式

- **顺序执行**：按照六阶段顺序依次执行，适合小型项目
- **迭代执行**：在大型项目中可分模块迭代执行
- **回溯修正**：发现偏差时，回溯到上游文档修正后重新执行

### 3.4 技能调用命令

本规范提供六个技能命令，可通过 `/opd:技能名` 格式调用：

**命令格式**：
```
/opd:<技能名> [参数]
```

**技能命令列表**：

| 调用命令 | 阶段 | 核心方法论 | 说明 |
|----------|------|------------|------|
| `/opd:office-hours` | 阶段一 | 六问题强迫思考 | 启动需求分析对话 |
| `/opd:plan-review` | 阶段二 | STRIDE威胁模型 | 设计与架构审核 |
| `/opd:develop` | 阶段三 | OWASP安全检查 | 代码开发与审查 |
| `/opd:qa` | 阶段四 | 真实浏览器QA | 端到端测试验证 |
| `/opd:ship` | 阶段五 | 发布管理 | 部署与交付 |
| `/opd:retro` | 阶段六 | 闭环回顾 | 运维与迭代优化 |

**使用示例**：

```
/opd:office-hours 我想开发一个任务管理应用
/opd:plan-review 审核当前架构设计
/opd:develop 实现用户登录功能
/opd:qa 测试登录流程
/opd:ship 发布v1.0版本
/opd:retro 分析本周运维数据
```

**链式调用**：

```
/opd:office-hours → /opd:plan-review → /opd:develop → /opd:qa → /opd:ship → /opd:retro
        ↑                                                                                    ↓
        └────────────────────────────── 闭环验证 ────────────────────────────────────────────┘
```

**技能文件位置**：`.qoder/skills/opd/`

## 四、质量保障

### 4.1 质量门禁

**阶段一门禁**：需求完整性检查
- 需求规格说明书完整
- 用户故事列表清晰
- 验收标准明确
- 项目计划合理

**阶段二门禁**：设计审查通过
- 架构设计合理
- 技术选型合适
- 接口设计清晰
- 数据库设计规范

**阶段三门禁**：代码质量达标
- 代码审查通过
- 单元测试覆盖率≥80%
- 代码规范符合标准
- 安全检查通过

**阶段四门禁**：测试通过
- 功能测试通过
- 性能测试达标
- 安全测试通过
- 测试报告完整

**阶段五门禁**：发布就绪
- 部署文档完整
- 发布说明清晰
- 用户手册完备
- 回滚方案明确

### 4.2 质量检查清单

**代码质量**：
- 命名规范
- 注释完整
- 代码复杂度
- 异常处理
- 安全漏洞

**功能质量**：
- 功能完整性
- 逻辑正确性
- 边界情况处理
- 用户体验

**性能质量**：
- 响应时间
- 资源占用
- 并发处理
- 扩展性

### 4.3 安全审计（OWASP + STRIDE）

开发过程中必须执行安全审计，采用 OWASP Top 10 和 STRIDE 威胁模型双重检查。

**OWASP Top 10 检查清单**：

| 编号 | 安全风险 | 检查要点 |
|------|----------|----------|
| A01 | 访问控制失效 | 身份认证、权限校验、会话管理 |
| A02 | 加密失败 |敏感数据加密、传输安全、密钥管理 |
| A03 | 注入攻击 | SQL注入、XSS、命令注入防护 |
| A04 | 不安全设计 | 业务逻辑漏洞、权限边界设计 |
| A05 | 安全配置错误 | 默认配置、错误信息泄露、未授权功能 |
| A06 | 脆弱组件 | 第三方依赖版本、已知漏洞检查 |
| A07 | 身份认证失败 | 密码策略、多因素认证、会话失效 |
| A08 | 数据完整性失败 | 输入验证、签名校验、防篡改 |
| A09 | 日志监控失败 | 安全日志、异常告警、审计追踪 |
| A10 | SSRF | 外部资源访问控制、URL验证 |

**STRIDE 威胁模型**：

| 威胁类型 | 英文 | 检查要点 |
|----------|------|----------|
| 仿冒 | Spoofing | 身份伪造、证书伪造、钓鱼防护 |
| 篡改 | Tampering | 数据完整性、防篡改机制 |
| 抵赖 | Repudiation | 操作日志、不可否认性证明 |
| 信息泄露 | Information Disclosure | 数据泄露、权限泄露、敏感信息暴露 |
| 拒绝服务 | Denial of Service | 资源耗尽防护、限流机制 |
| 权限提升 | Elevation of Privilege | 越权访问、权限边界 |

**安全审计时机**：
- 阶段二设计完成后：STRIDE 威胁模型分析
- 阶段三代码完成后：OWASP Top 10 代码扫描
- 阶段四测试阶段：渗透测试验证
- 阶段五发布前：最终安全检查报告

## 五、最佳实践

### 5.1 项目管理

- **需求管理**：明确需求优先级，避免需求蔓延
- **时间管理**：合理估算时间，设置里程碑
- **风险管理**：识别潜在风险，制定应对措施
- **资源管理**：有效分配工作任务，避免过载

### 5.2 技术实践

- **代码规范**：统一编码标准，提高代码可读性
- **版本控制**：使用Git进行代码管理
- **测试驱动**：先写测试后实现功能
- **持续集成**：自动化测试和部署

### 5.3 协作技巧

- **清晰沟通**：明确任务描述和期望
- **及时反馈**：快速响应问题和调整方向
- **适度干预**：在关键节点提供指导
- **持续学习**：积累和分享最佳实践
