|---------|------|---------|

### 2.2 L1：代码级质量保障

#### 2.2.1 职责定义

| 项目 | 内容 |
|------|------|
| **负责角色** | SEA（软件编码员工） |
| **质量活动** | 编码规范遵循、静态分析、自测验证 |
| **执行时机** | 代码编写过程中、提交前 |
| **输出结果** | 符合规范的代码、自检报告 |

#### 2.2.2 检查内容

```yaml
L1质量检查清单:
  编码规范:
    □ 命名规范：变量、函数、类命名清晰有意义
    □ 代码格式：缩进、空格、换行符合规范
    □ 注释规范：关键逻辑有注释说明
    □ 代码结构：逻辑清晰，层次分明
    
  静态分析:
    □ 无编译错误
    □ 无编译警告
    □ 无静态分析工具报告的问题
    □ 无代码坏味道
    
  自测验证:
    □ 功能逻辑正确
    □ 边界条件处理
    □ 异常情况处理
    □ 基本功能可用
    
  文档完整:
    □ 代码注释完整
    □ API文档更新
    □ 变更说明记录
```

#### 2.2.3 执行规范

```
SEA代码提交前自检流程:

1. 编码完成
   └──▶ 2. 编码规范检查
        └──▶ 3. 静态分析
             └──▶ 4. 自测验证
                  └──▶ 5. 文档更新
                       └──▶ 6. 提交代码审查

任何一步不通过，返回修改后重新检查
```

  单元测试:
    □ 测试覆盖率 ≥ 80%
    □ 测试用例有效
    □ 所有测试通过
    □ 边界场景覆盖
    
  接口测试:
    □ 接口定义符合设计

    □ 设计文档符合
    □ 编码规范符合
    □ 安全规范符合
    □ 文档规范符合

### 3.2 门禁详细定义

#### 3.2.1 需求完整性门禁

```yaml
门禁名称: 需求完整性门禁
触发时机: 需求阶段结束，进入设计阶段前
把关角色: HD（人类开发者）

通过条件:
  必须项:
    - 所有业务需求已识别并记录
    - 功能需求列表完整
    - 非功能需求已定义
    - 需求优先级已确定
    - 验收标准已明确
    
  推荐项:
    - 用户故事已编写
    - 需求依赖关系已梳理
    - 风险已识别

检查方式:
  - 文档审核
  - 需求评审会议
  - HD签字确认

不通过处理:
  - 返回SDA补充需求分析
  - 明确缺失项和完成期限
```

#### 3.2.2 设计评审门禁

```yaml
门禁名称: 设计评审门禁
触发时机: 设计阶段结束，进入开发阶段前
把关角色: SRA（软件审核员工）

通过条件:
  必须项:
    - 架构设计文档完整
    - 技术选型有理有据
    - 接口定义完整清晰
    - 数据模型设计合理
    - 安全方案已设计
    
  推荐项:
    - 性能方案已设计
    - 部署方案已规划
    - 技术风险已评估

检查方式:
  - 文档审核
  - 架构评审
  - SRA审核报告

不通过处理:
  - 返回SDA修改设计
  - 记录问题和修改建议
  - 重新审核
```

#### 3.2.3 代码质量门禁

```yaml
门禁名称: 代码质量门禁
触发时机: 代码提交时、模块开发完成时
把关角色: SRA（软件审核员工）

通过条件:
  必须项:
    - 代码编译无错误
    - 单元测试全部通过
    - 代码覆盖率 ≥ 80%
    - 无严重静态分析问题
    - 代码审查通过
    
  推荐项:
    - 无重要静态分析问题
    - 代码复杂度合理
    - 无重复代码警告

检查方式:
  - 自动化构建
  - 自动化测试
  - 静态代码分析
  - 代码审查

不通过处理:
  - 返回SEA修改代码
  - 记录问题和修改建议
  - 重新提交审核
```

#### 3.2.4 测试通过门禁

```yaml
门禁名称: 测试通过门禁
触发时机: 测试阶段结束，进入部署阶段前
把关角色: STA（软件测试员工）+ SRA（软件审核员工）

通过条件:
  必须项:
    - 功能测试全部通过
    - 测试通过率 ≥ 95%
    - 无未修复的严重缺陷
    - 性能指标达标
    - 安全测试通过
    
  推荐项:
    - 无未修复的重要缺陷
    - 自动化测试覆盖核心功能

检查方式:
  - 测试报告审核
  - 缺陷统计检查
  - HD验收测试

不通过处理:
  - 返回SEA修复缺陷
  - 执行回归测试
  - 重新评估发布
```

#### 3.2.5 发布就绪门禁
