# 一人AI开发工程

**One-Person AI Development Project**

---

## 项目简介

本项目是一套完整的软件开发规范体系，旨在让**一个人**通过**六阶段标准化流程**完成完整的软件项目开发。

### 核心理念

一人AI开发模式下，开发者通过六个标准化阶段完成项目：

```mermaid
graph TB
    P1[阶段一: 需求与规划]
    P2[阶段二: 设计与架构]
    P3[阶段三: 开发与实现]
    P4[阶段四: 测试与验证]
    P5[阶段五: 部署与交付]
    P6[阶段六: 运维与迭代]
    HD[HD: 人类开发者]
    
    P1 --> P2 --> P3 --> P4 --> P5 --> P6
    P1 --> HD
    P6 --> HD
```

### 六阶段流程

| 阶段 | 名称 | 占比 | 核心任务 | 执行约束 |
|------|------|------|----------|----------|
| 阶段一 | 需求与规划 | 10-15% | 需求分析、项目计划、验收标准 | - |
| 阶段二 | 设计与架构 | 15-20% | 架构设计、技术选型、接口设计 | - |
| 阶段三 | 开发与实现 | 40-50% | 代码开发、单元测试、代码审查 | ⚠️ 长时间执行，不可中断 |
| 阶段四 | 测试与验证 | 15-20% | 测试设计、测试执行、缺陷管理 | ⚠️ 长时间执行，不可中断 |
| 阶段五 | 部署与交付 | 5-10% | 部署配置、发布说明、用户手册 | - |
| 阶段六 | 运维与迭代 | 持续 | 系统监控、问题分析、迭代优化 | - |

> ⚠️ **重要提醒：dev 和 qa 阶段长时间执行不可中断**
> - **阶段三（开发与实现）**：耗时最长，中断会导致代码状态不一致、上下文丢失
> - **阶段四（测试与验证）**：涉及真实浏览器测试，中断会导致会话丢失、需重新测试
> - **建议**：为这两个阶段预留充足的完整工作时段

## OPD技能包使用

### 技能包介绍

OPD技能包是一套可导入Qoder使用的标准化开发流程技能，包含六个阶段技能和文档模板。

### 调用命令

| 阶段 | 命令 | 说明 |
|------|------|------|
| 阶段一 | `/opd:req` | 需求分析（六问题强迫思考） |
| 阶段二 | `/opd:ds` | 设计与架构（STRIDE威胁模型） |
| 阶段三 | `/opd:dev` | 开发实现（OWASP安全检查）⚠️ |
| 阶段四 | `/opd:qa` | 测试验证（真实浏览器QA）⚠️ |
| 阶段五 | `/opd:ship` | 部署交付 |
| 阶段六 | `/opd:retro` | 运维迭代 |

### 链式调用

```
/opd:req → /opd:ds → /opd:dev → /opd:qa → /opd:ship → /opd:retro
```

```mermaid
graph LR
    P1[阶段一 需求与规划]

```

**核心方法论**：

| 阶段 | 核心方法论 |
|------|------------|
| 阶段一 | **六问题强迫思考**：需求分析前必须回答6个关键问题 |
| 阶段二 | **STRIDE威胁模型**：安全威胁分析与架构设计同步进行 |
| 阶段三 | **OWASP安全检查**：代码审查时执行OWASP Top 10安全扫描 ⚠️ |
| 阶段四 | **真实浏览器QA**：使用 Playwright 进行端到端测试 ⚠️ |

> ⚠️ 标记的阶段需要长时间连续执行，不可中断

## 质量保障体系

### 四层质量模型

```mermaid
graph TB
    L4[第四层 项目级质量]
    L3[第三层 系统级质量]
    L2[第二层 模块级质量]
    L1[第一层 代码级质量]
    
    L4 --> L3 --> L2 --> L1
```

### 质量门禁

| 门禁名称 | 检查项 | 通过标准 |
|---------|-------|---------|
| 需求完整性门禁 | 需求文档完整性 | 100%完成 |
| 设计评审门禁 | 架构设计评审 | 审核通过 |
| 代码质量门禁 | 代码审查、单元测试 | 覆盖率≥80% |
| 测试通过门禁 | 功能测试、性能测试 | 通过率100% |
| 发布就绪门禁 | 部署验证、文档检查 | 全部通过 |

## 链式调用机制

### 文档流转

六阶段开发流程采用**链式调用**设计：每个阶段的产出文档自动成为下一阶段的输入，形成闭环。

```mermaid
graph TB
    D1[需求规格说明书]
    D2[架构设计文档]
    D3[代码实现]
    D4[测试用例]
    D5[发布说明]
    D6[迭代计划]
    VC[验收标准]
    
    D1 --> D2 --> D3 --> D4 --> D5 --> D6
    D1 --> VC
    D6 --> VC
```

### 执行模式

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| 顺序执行 | 按照六阶段顺序依次执行 | 小型项目 |
| 迭代执行 | 分模块迭代执行 | 大型项目 |
| 回溯修正 | 发现偏差时回溯上游文档修正 | 问题修复 |

## 常见问题

### Q1：一人AI开发模式是什么？

一人AI开发模式是一种标准化的软件开发流程，让一个人通过六个阶段（需求与规划、设计与架构、开发与实现、测试与验证、部署与交付、运维与迭代）完成完整的软件项目开发。

### Q2：链式调用机制有什么好处？

链式调用机制确保每个阶段的产出文档自动成为下一阶段的输入，形成闭环，避免信息丢失和需求偏离。

### Q3：什么是六问题强迫思考？

需求分析阶段采用"六问题强迫思考"方法论，在编写需求文档前必须回答六个关键问题，挖掘真实需求，避免伪需求。

### Q4：如何保证开发质量？

1. 提供详细的需求和设计文档
2. 执行 OWASP + STRIDE 双重安全审计
3. 使用真实浏览器进行端到端测试
4. 通过五个质量门禁进行阶段验收
