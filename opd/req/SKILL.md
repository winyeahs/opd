# 阶段一：需求与规划

**执行时间占比**：10-15%

## 核心任务

需求分析、项目计划、验收标准制定

## 核心方法论：六问题强迫思考

在编写需求文档前必须回答以下六个问题：

| 问题编号 | 问题内容 | 目的 |
|----------|----------|------|
| Q1 | 用户现在的痛点是什么？具体例子，不是假设 | 挖掘真实需求，避免伪需求 |
| Q2 | 用户目前是怎么解决这个问题的？ | 了解现状，找到改进空间 |
| Q3 | 如果不解决这个问题，后果是什么？ | 确认需求优先级和紧迫性 |
| Q4 | 你提出的功能真的解决了痛点吗？还是只是"听起来不错"？ | 挑战方案有效性 |
| Q5 | 最小可行方案是什么？明天能发布的是什么？ | 确定MVP范围，避免过度设计 |
| Q6 | 这个需求的假设前提是什么？假设成立吗？ | 检验需求根基，避免在错误假设上建设 |

**执行要点**：
- 每个问题必须给出具体答案，不得跳过
- 答案要有具体例子支撑，拒绝泛泛而谈
- 如果某个问题无法回答，说明需求分析不够深入
- 通过六问题后，需求规格说明书才能进入下一阶段

## 交付物

生成以下文档到项目根目录的 `docs/阶段一-需求与规划/` 目录：

**参考模板**：技能包内 `templates/阶段一需求与规划/` 目录

1. **需求规格说明书.md** - 必需
   - 用户痛点分析
   - 功能需求列表
   - 非功能需求
   - 用户故事

2. **前端UI设计说明书.md** - 必需
   - 页面布局设计
   - 交互流程设计
   - UI组件设计

3. **验收标准.md** - 必需
   - 功能验收标准
   - 性能验收标准
   - 用户体验验收标准

4. **项目计划.md** - 必需
   - 里程碑规划
   - 时间估算
   - 资源分配

5. **风险清单.md** - 必需
   - 风险识别
   - 风险等级评估
   - 应对措施

## 执行流程

1. **明确目标**：与用户进行需求对话，理解项目背景和目标
2. **六问题思考**：逐一回答六个核心问题，记录具体答案
3. **需求分析**：基于六问题答案，提炼功能需求和非功能需求
4. **产出文档**：生成五份核心文档
5. **质量门禁检查**：验证文档完整性和合理性

## 质量门禁

**阶段一门禁**：需求完整性检查
- [ ] 需求规格说明书完整
- [ ] 用户故事列表清晰
- [ ] 验收标准明确
- [ ] 项目计划合理

## 与上下游关系

- **上游**：无（这是第一阶段）
- **下游**：阶段二（ds）读取 `docs/阶段一-需求与规划/` 目录的交付物
- **回溯**：最终产出需回溯验收标准，确保不偏离原始需求

## 使用示例

```
/opd:req 我想开发一个任务管理应用
/opd:req 需要做一个在线教育平台
/opd:req 帮我规划一个电商系统
```

## 注意事项

- 避免需求蔓延，明确需求优先级
- 确保每个需求都有明确的验收标准
- 风险识别要全面，应对措施要具体
- 项目计划要留有缓冲时间
