# 阶段二：设计与架构

**执行时间占比**：15-20%

## 核心任务

架构设计、技术选型、接口设计

## 核心方法论：STRIDE威胁模型

在设计阶段必须执行STRIDE威胁分析：

| 威胁类型 | 英文 | 检查要点 |
|----------|------|----------|
| 仿冒 | Spoofing | 身份伪造、证书伪造、钓鱼防护 |
| 篡改 | Tampering | 数据完整性、防篡改机制 |
| 抵赖 | Repudiation | 操作日志、不可否认性证明 |
| 信息泄露 | Information Disclosure | 数据泄露、权限泄露、敏感信息暴露 |
| 拒绝服务 | Denial of Service | 资源耗尽防护、限流机制 |
| 权限提升 | Elevation of Privilege | 越权访问、权限边界 |

**执行要点**：
- 对每个关键功能模块执行STRIDE分析
- 识别潜在威胁并设计应对措施
- 将安全设计融入架构文档

## 执行流程

1. **读取上游文档**：读取阶段一的需求规格说明书等文档
2. **架构设计**：设计系统整体架构，选择技术栈
3. **详细设计**：设计前后端技术方案、接口、数据库
4. **安全分析**：执行STRIDE威胁模型分析
5. **设计审核**：审核设计合理性，生成审核报告
6. **质量门禁检查**：验证设计文档完整性

## 质量门禁

**阶段二门禁**：设计审查通过
- [ ] 架构设计合理
- [ ] 技术选型合适
- [ ] 接口设计清晰
- [ ] 数据库设计规范
- [ ] STRIDE威胁分析完成

## 与上下游关系

- **上游**：读取 `docs/阶段一-需求与规划/` 目录的交付物
- **下游**：阶段三（dev）读取 `docs/阶段二-设计与架构/` 目录的交付物
- **链式调用**：确保设计不偏离需求定义

## 注意事项

- 架构设计要考虑可扩展性和可维护性
- 技术选型要权衡团队能力和项目需求
- 接口设计要遵循RESTful规范
- 数据库设计要考虑性能和数据一致性
- 安全分析要全面，不能遗漏关键威胁
