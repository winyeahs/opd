# 阶段六：运维与迭代

**执行时间占比**：持续

## 核心任务

系统监控、问题分析、迭代优化

## 核心方法论：闭环回顾

运维阶段采用闭环回顾方法论，形成持续改进循环：

### 回顾循环

```
数据收集 → 问题分析 → 优化方案 → 实施改进 → 效果验证 → 数据收集
```

### 回顾要点

1. **数据驱动**：基于实际运维数据进行分析
2. **问题导向**：聚焦影响用户体验的关键问题
3. **闭环验证**：优化后验证效果，确保改进有效
4. **持续迭代**：不断发现问题、解决问题、优化系统

## 执行流程

1. **数据收集**：收集系统运行数据、用户反馈、性能指标
2. **状态评估**：评估系统运行状态，识别异常和问题
3. **问题分析**：分析问题根因，评估影响程度
4. **制定方案**：制定优化方案和迭代计划
5. **实施改进**：执行优化方案，修复问题
6. **效果验证**：验证优化效果，确保改进有效
7. **生成报告**：生成运维报告和优化建议
8. **闭环迭代**：形成下一迭代周期的输入

## 运维监控指标

### 系统指标

- CPU使用率
- 内存使用率
- 磁盘使用率
- 网络流量

### 应用指标

- 请求响应时间
- 请求成功率
- 错误率
- 并发连接数

### 业务指标

- 用户活跃度
- 功能使用率
- 转化率
- 用户留存率

## 问题分级标准

| 级别 | 定义 | 处理时限 |
|------|------|----------|
| P0 | 系统不可用，严重影响用户 | 立即处理 |
| P1 | 核心功能异常，影响部分用户 | 24小时内处理 |
| P2 | 非核心功能异常，用户体验下降 | 72小时内处理 |
| P3 | 优化建议，不影响正常使用 | 下次迭代处理 |

## 与上下游关系

## 注意事项

- 数据收集要全面，包括系统和业务指标
- 问题分析要深入，找到根本原因而非表面现象
- 优化方案要可执行，有明确的时间安排和责任人
- 效果验证要客观，用数据说话
- 闭环回顾要形成持续改进的文化
- 迭代计划要与验收标准对齐，确保不偏离原始需求
